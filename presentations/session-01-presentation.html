üìù What is a Log?

Definition

A log is a record of events or activities that happen on a computer or network device. Think of it as a digital diary that automatically records everything that happens.

üîç Real-World Analogy

Just like a security guard writes down everyone who enters a building, computer systems write down (log) every important event that occurs.

Key Characteristics of Logs:

üìÖ Timestamped
Shows exactly when events occurred
üìç Detailed
Records who, what, where, and how
üîÑ Automatic
Created without human intervention
üíæ Persistent
Stored for future reference

üõ°Ô∏è Why Logs Are Critical in Cybersecurity

üö® Detection & Response

Logs help cybersecurity professionals detect suspicious activity, investigate security incidents, and understand how an attack happened.

Key Benefits:

  • Early Warning System: Detect attacks as they happen
  • Forensic Investigation: Understand what happened after an incident
  • Compliance: Meet regulatory requirements
  • Performance Monitoring: Track system health and performance
  • Legal Evidence: Provide proof in legal proceedings

üíº Business Impact

Without proper logging, organisations are flying blind - they can't see attacks coming, can't investigate incidents properly, and can't prove what happened in court.

üíæ Where Are Logs Stored?

Logs are usually stored on the device where the activity happened - like a computer, server, firewall, or switch. Some organisations also send logs to a central system for easier monitoring.

Storage Locations:

  • Local Storage: On the device that generated the log
  • Network Storage: On dedicated log servers
  • Cloud Storage: In cloud-based logging services
  • SIEM Systems: Security Information and Event Management platforms

üîÑ centralised vs. Distributed Logging

centralised: All logs sent to one location for easy analysis
Distributed: Logs stored locally on each device

üí° Best Practice

Use centralised logging for better security monitoring and easier analysis, but maintain local backups for redundancy.

‚öñÔ∏è Logs as Digital Evidence

üìã Legal Admissibility

Yes, logs are often used as digital evidence in investigations to show what actions were taken and by whom.

Requirements for Legal Evidence:

  • Integrity: Logs must be tamper-proof and unmodified
  • Chain of Custody: Clear record of who handled the evidence
  • Authenticity: Proof that logs are genuine and accurate
  • Completeness: All relevant log data must be preserved
  • Accessibility: Logs must be in a readable format

üîí Protection Requirements

To be used as evidence, logs must be protected from tampering and unauthorized access to ensure their integrity and usefulness in investigations.

üéØ Key Logging Points:

üö® Case Study: Ransomware Attack Analysis

üìñ The Scenario

A small company suffered a ransomware attack. The breach originated from PC-A where a user downloaded a malicious PDF from a phishing email, which installed a botnet client and established a reverse shell connection.

üîç Log Analysis by Device:

üíª PC-A (Patient Zero)

  • Antivirus logs: Malware detection alerts
  • Browser history: Phishing site visits
  • Email logs: Phishing email receipt
  • System event logs: Suspicious process execution
  • Application logs: PDF reader activity

üóÑÔ∏è File Server (Target)

  • Authentication logs: Compromised credential usage
  • File access logs: Unusual after-hours activity
  • Audit logs: File encryption activities
  • System logs: Remote access evidence

üîå Network Infrastructure Logs

üõ°Ô∏è Firewall Logs

  • Traffic logs: Outbound reverse shell connections
  • IDS/IPS logs: Malware signature alerts
  • Connection logs: Unusual ports/protocols
  • Blocked connection attempts

üîÄ Network Switch Logs

  • Port activity logs: High traffic volumes
  • MAC address tables: Device mappings
  • VLAN logs: Segmentation bypass attempts
  • Spanning tree protocol changes

üéØ Investigation Strategy

Start with the firewall logs to identify suspicious outbound connections, then trace back through switch logs to identify the source device, and finally examine endpoint logs for the attack vector.

üèõÔ∏è The Essential Eight Framework

üìã What is the Essential Eight?

The Essential Eight is a set of eight cybersecurity strategies recommended by the Australian Cyber Security Centre (ACSC) to help organisations protect their systems from cyber threats.

üîç Logging-Related Strategies:

  • Restrict Administrative Privileges: Requires extensive logging to monitor admin access
  • Multi-Factor Authentication: Needs logs to track authentication events
  • Application Control: Logs software execution and blocking events
  • Patch Applications & OS: Logs update activities and vulnerabilities

üìä Maturity Model (Levels 1-3)

The Essential Eight Maturity Model shows how well an organisation has implemented each strategy. Higher levels require more thorough logging and monitoring.

üìà Logging Requirements for Essential Eight

üéØ Why Logging Matters for Admin Privileges

Logs help track who accessed admin accounts, when, and what actions were taken - making it easier to detect unauthorised or risky behaviour.

üöÄ Software Execution

üåê Network Traffic

üîÑ Patch Activities
Update installations, failures

üõ†Ô∏è Log Management Best Practices

‚è∞ Log Retention

Log retention refers to how long logs are stored. Keeping logs for a sufficient period (e.g., 12 months) helps with long-term investigations and compliance.

üè¢ centralised Logging

centralised logging means sending logs from multiple devices to a single system for easier monitoring, analysis, and alerting.

üéØ Best Practices:

  • Standardize formats: Use consistent log formats across systems
  • Implement real-time monitoring: Set up alerts for suspicious activities
  • Regular backups: Ensure logs are backed up and recoverable
  • Access controls: Restrict who can view and modify logs
  • Regular reviews: Periodically analyze logs for trends

üîç Using Logs for Threat Detection

ü¶† Malware & Ransomware Detection

Logs can show unusual behaviour like unexpected file access, strange network connections, or unauthorised login attempts - clues that malware may be active.

üö® Warning Signs in Logs:

  • Unusual login times: Access during off-hours
  • Failed authentication spikes: Potential brute force attacks
  • Abnormal file access: Mass file encryption or deletion
  • Suspicious network traffic: Connections to unknown external IPs
  • Privilege escalation: Users gaining unexpected admin rights
  • Process anomalies: Unknown applications running

‚ö° Real-Time Detection

Modern SIEM systems can analyze logs in real-time and automatically alert security teams when suspicious patterns are detected.

üöë Logs in Incident Response

üìÖ Timeline Reconstruction

Logs provide a timeline of events, helping analysts understand how an attack happened and what systems were affected.

üîç Incident Response Process:

  • Detection: Logs trigger alerts about suspicious activity
  • Investigation: Analyze logs to understand the scope
  • Containment: Use log data to isolate affected systems
  • Eradication: Remove threats based on log evidence
  • Recovery: Monitor logs during system restoration
  • Lessons Learned: Review logs to improve defenses

‚öñÔ∏è Forensic Analysis

Logs serve as digital fingerprints, providing evidence of what happened, when it occurred, and who was involved in a security incident.
